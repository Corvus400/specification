# CMS 要件定義

本書は、架空医薬品・疾患リファレンスの管理機能を操作する **CMS（管理画面）の機能・非機能・UI/UX 要件 (WHAT)** を定義する。実現手段 (HOW)（フレームワーク構成・コンポーネント設計・状態管理ライブラリ・スタイリング手法等）は本書では規定せず、実装フェーズで別途設計する。

関連文書:

- アクセス制御（管理面の到達経路・外部秘匿・認証/認可・CORS）は `13_秘匿アクセス制御要件定義.md` で定義する。CMS はその管理面に接続するクライアントであり、認証・CORS・到達経路は同書を前提とする。
- CMS が呼び出す CRUD WebAPI の入出力契約・画像アップロード・競合検出は `14_CRUD_WebAPI要件定義.md` で定義する。本書はその API を**操作するクライアント側の要件**を規定し、サーバ契約を再定義しない。
- 編集対象フィールドの意味論・値域・必須/任意の別は既存モデル詳細 `03_医薬品モデル詳細.md` / `04_疾患モデル詳細.md` に従う。本書はそれらを再定義せず、編集 UI が満たすべき要件のみを規定する。

前提:

- CMS は管理者がローカル環境で利用する管理ツールであり、公開アプリ（一般利用者向け）とは別の成果物である。
- CMS が操作する API はローカル限定の管理面に存在し、外部からは秘匿される（`13_` §2・§3）。CMS 自身もローカル環境で動作する。
- CMS はサーバの状態（drug / disease レコード、アップロード画像）を取得・変更する唯一の管理 UI であり、編集結果は実 DB に永続化される（`14_` §10）。

---

## 1. 目的

- drug / disease に対する Create / Read / Update / Delete を、管理者が GUI から安全かつ正確に実行できるようにする。
- 多数のフィールド（nested 構造・enum・配列・関連 ID を含む）を、入力可否・必須/任意・値域の制約に沿って編集できるフォーム体験を提供する。
- drug 画像のアップロードを GUI から行えるようにする。
- サーバが検出する検証エラー・参照整合性違反・編集競合を、管理者が理解し修正できる形で提示する。
- ローカル専用・管理者限定のツールとして、`13_` の秘匿・認証要件と整合した接続・認証フローで動作する。

---

## 2. 対象スコープ・用語

| 用語 | 定義 |
|---|---|
| CMS | drug / disease の CRUD を操作する、ローカル限定の管理画面。本書の対象成果物。 |
| 管理面 | CMS が接続する、ローカル限定・外部秘匿の到達経路（`13_` §2 が定義）。 |
| 編集対象リソース | drug（`drug_NNNN`）/ disease（`disease_NNNN`）。 |
| コンテンツフィールド | クライアントが編集できるフィールド（`14_` §4.1 CRUD-1。サーバ管理フィールド `14_` §3 を除く）。 |

スコープ:

- 本書が対象とするのは CMS（クライアント）の要件である。サーバ側の CRUD 契約・到達制御・認証実装は `13_` / `14_` が SSOT であり、本書では参照に留める。
- CMS の実装技術選定・コンポーネント設計・ビルド構成（HOW）は本書の対象外（§6 に技術前提のみ記載）。
- 公開アプリ（一般利用者向け UI）は本書の対象外。

## 3. 機能要件（acceptance criteria）

各要件は検証可能な受け入れ基準として記述する。API の入出力契約・ステータスコードは `14_` を参照し、本書では UI が満たすべき振る舞いを規定する。

### 3.1 一覧・検索・参照

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CMS-1 | リソース一覧 | drug / disease それぞれの一覧を表示できる。一覧は既存の読み取り一覧契約（`14_` CRUD-4：`items` とページング `page` / `pageSize` / `totalPages` / `totalCount`）に基づき、CRUD で編集されたレコードも反映される。 |
| CMS-2 | 検索・フィルタ・ソート | 既存一覧 API が提供するキーワード検索・フィルタ・ソート（`14_` CRUD-4 が変更しないとする条件群）を UI から指定して絞り込めること。提供される条件の範囲を超える独自検索は要求しない。 |
| CMS-3 | ページング | 一覧のページ送りができ、現在ページ・総件数を確認できること。 |
| CMS-4 | 詳細参照 | 一覧から個別リソースの詳細を開き、全フィールド（nested 構造を含む）を確認できること。詳細は既存詳細契約（`14_` CRUD-5）に基づく。 |

### 3.2 作成・編集フォーム

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CMS-5 | 新規作成 | コンテンツフィールド（`14_` CRUD-1）を入力して新規 drug / disease を作成できる。サーバ管理フィールド（`14_` §3：`id` / `revisedAt` / `imageUrl` / `disclaimer` / タイムスタンプ）は**入力欄を設けず**、サーバ採番・サーバ設定値に委ねる。作成成功後、採番された `id` を含む結果を確認できる。 |
| CMS-6 | 編集 | 既存リソースのコンテンツフィールドを編集できる。更新方式（全置換 / 部分更新）は `14_` §4.4 のサーバ契約に従い、UI はそのいずれかで保存を成立させる。外部公開 `id` は編集不可として提示する（`14_` CRUD-12）。 |
| CMS-7 | nested 構造の編集 | モデルが持つ nested 構造（drug: 組成・用法用量・副作用・薬物動態・包装・参考文献ほか / disease: 症状・診断基準・治療・疫学・重症度ほか）を、構造を保ったまま編集できる UI を提供する。各 nested セクション・配列項目の追加・削除・並びを操作できること。フィールドの意味論・構成は `03_` / `04_` に従う。 |
| CMS-8 | enum 選択 | enum 型フィールド（drug: 剤形・規制区分・投与経路 / disease: ICD-10 章・慢性度・診療科）は、モデルが定義する許容値（`14_` CRUD-20）からの選択 UI とし、自由入力で未知値を送出させない。複数選択を取る enum（規制区分・診療科等）は複数選択 UI とする。 |
| CMS-9 | 配列フィールドの編集 | 文字列配列・段落配列等の繰り返しフィールドを、項目単位で追加・編集・削除できること。 |
| CMS-10 | 必須/任意の明示 | 必須フィールド（`14_` CRUD-19、`03_`/`04_` が必須とするもの）と任意フィールドを UI 上で区別して提示し、必須欠落は保存前または保存時に判別できること。 |

### 3.3 関連リソース（relatedIds）の編集

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CMS-11 | 関連 ID 編集 | drug の `relatedDiseaseIds`、disease の `relatedDrugIds` / `relatedDiseaseIds` を編集できる。関連先は実在するリソースから選択する UI とし、存在しない ID を手入力で混入させない設計を要件とする（サーバはダングリング参照を拒否する：`14_` CRUD-21）。 |
| CMS-12 | 自己参照の抑止 | 編集中リソース自身を関連 ID に選べないこと（サーバは自己参照を拒否する：`14_` CRUD-22）。UI でも選択候補から除外または抑止する。 |
| CMS-13 | 関連先の可読表示 | 関連 ID は ID だけでなく対象リソースの名称等で識別できる形で提示し、誤った関連付けを防ぐこと。 |

### 3.4 削除

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CMS-14 | 削除操作 | drug / disease を削除できる。削除は破壊的操作のため、実行前に確認の手順を介して二段階で確定する（誤操作防止）。 |
| CMS-15 | 被参照時の競合提示 | 他リソースから参照されているリソースの削除はサーバが拒否する（`14_` CRUD-17、409）。CMS はこの競合をユーザに提示し、「参照を外してから削除する」運用に導く（どのリソースから参照されているかが分かる手掛かりを示すことが望ましい）。 |
| CMS-16 | 画像の連動 | drug 削除時、サーバ側でアップロード画像も到達不能になる（`14_` CRUD-18）。CMS は削除後にその drug の画像参照を残さないこと。 |

### 3.5 画像アップロード（drug のみ）

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CMS-17 | アップロード UI | drug 詳細・編集画面から画像をアップロードできる。対象形式は PNG、サイズ上限を超える/非対応形式はサーバが拒否する（`14_` CRUD-28）。CMS はアップロード前または失敗時に形式・サイズの不適合をユーザに伝える。 |
| CMS-18 | プレビューと反映 | アップロード対象/結果の画像をプレビューでき、アップロード後の配信結果（`14_` CRUD-26・CRUD-27 の `imageUrl` 整合）が UI に反映されること。 |
| CMS-19 | サイズ指定の整合 | 既存配信のサイズ指定（`S` / `M` / `Original`、`14_` CRUD-30）と整合した形で画像を確認できること。 |
| CMS-20 | 疾患に画像なし | disease には画像アップロード UI を設けない（`14_` §2：画像を持つのは drug のみ）。 |

## 4. 非機能要件（acceptance criteria）

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CMS-21 | ローカル専用動作 | CMS はローカル環境で動作し、ローカル限定の管理面（`13_` §2）にのみ接続する。公開面（外部公開の読み取り経路）を書き込みに用いない。 |
| CMS-22 | 認証フロー | CMS は `admin` scope 付き JWT をローカル環境で取得し（取得手段は `13_` SEC-6 が提供）、保持し、すべての CRUD/管理リクエストに付与する。認証なし/不正トークンではサーバが 401、scope 不足では 403 を返す（`13_` SEC-4・SEC-5）。 |
| CMS-23 | 認証失敗時の挙動 | サーバから 401（未認証・トークン失効）を受けた場合、CMS は再認証（トークン再取得）に導き、403（権限不足）とは区別して提示する。401/403/429（throttle）を取り違えて表示しないこと。 |
| CMS-24 | トークンの安全な取り扱い | CMS は JWT を URL・ログ・第三者に観測可能な場所に露出させない。署名 secret 等の秘密情報をクライアントコード・公開リポジトリに埋め込まない（`13_` §6・SEC-6）。 |
| CMS-25 | cross-origin 前提 | CMS の origin と管理面の接続先は異なる（cross-origin）。管理面が許可する CORS（`13_` SEC-10：必要メソッドと認証ヘッダを許可、ワイルドカード不可）の範囲で動作すること。CMS は許可された接続先のみを参照し、接続先の具体値は private な構成（環境変数等）から解決する（`13_` §6）。 |
| CMS-26 | サーバ状態の整合 | CMS が表示するデータはサーバの現在状態と整合させる。作成・更新・削除の後、関連する一覧・詳細の表示を最新化し、古い表示を残さないこと。 |
| CMS-27 | 編集競合の処理 | サーバが楽観的同時実行制御で編集競合を検出する（`14_` §7、409/412）。CMS は更新時に既知の版を提示し、競合を受けた場合は上書きを強行せず、最新状態の取得・再編集に導く（編集消失を防ぐ）。 |
| CMS-28 | エラーの可読提示 | サーバが返す RFC 9457 problem+json（`14_` §9）を、ユーザが理解・対処できる形で提示する。フィールド単位の検証違反（`errors[]`）は該当フィールドに対応づけて表示し、どの入力をどう直せばよいか分かること。 |
| CMS-29 | 破壊操作の保護 | 削除・全置換等の不可逆/影響の大きい操作は、誤操作で即時確定しないこと（CMS-14 と整合）。 |

## 5. UI/UX 要件（acceptance criteria）

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CMS-30 | 情報設計 | drug / disease の一覧 → 詳細 → 編集という階層が明快で、リソース種別・現在位置・実行可能な操作が把握できること。多数フィールドを持つ編集画面は、モデルの構造（`03_`/`04_` のセクション構成）に沿って整理し、目的のフィールドに辿り着けること。 |
| CMS-31 | 画面遷移 | 一覧・詳細・新規作成・編集・削除確認の遷移が一貫し、操作の前後で文脈（対象リソース・編集中か否か）を見失わないこと。未保存の変更を破棄しうる遷移ではその旨を喚起すること。 |
| CMS-32 | 入力フィードバック | 保存・削除・アップロードの成功/失敗が明確に伝わること。検証エラーは入力箇所に紐づけて即座に分かる形で示すこと（CMS-28 と整合）。処理中・完了の状態が分かること。 |
| CMS-33 | アクセシビリティ | フォーム要素にラベルが対応づき、キーボードのみで主要操作（一覧操作・フォーム入力・保存・削除）が完結すること。エラー・状態の通知が支援技術にも伝わること。色のみに依存せず情報を伝えること。十分なコントラストを確保すること。 |
| CMS-34 | レスポンシブ | 管理者のデスクトップ利用を主対象としつつ、ウィンドウ幅の変化でレイアウトが破綻せず操作可能であること。 |
| CMS-35 | 一貫性 | 用語・操作・レイアウトのパターンを画面間で一貫させ、学習コストと誤操作を抑えること。 |

## 6. 技術前提

- CMS は React による SPA を前提とする。ローカル管理ツールであり SSR / SEO 要件がないため、ビルド基盤は Vite による SPA 構成を推奨する。
- 上記は方向性の前提であり、プロジェクト構造・状態管理・データ取得・スタイリング等の具体構成（HOW）は実装フェーズで設計する。本書では規定しない。

---

## 7. 公開ドキュメントの情報漏洩防止（運用要件）

CMS リポジトリ（README・設定例・コード）を含む公開成果物には、`13_` §6 と同じ制約を適用する。

- **ライブ環境の正確な接続先を書かない**: 管理面のポート番号、edge tunnel 識別子、credentials ファイルのパス。接続先は private な構成（環境変数・非公開設定）で SSOT 管理し、公開物には概念レベル（「ローカル専用経路に接続する」等）の記述に留める。
- **秘密情報を書かない**: JWT 署名 secret、DB 接続情報、各種 API トークン、秘密鍵。
- **個人情報を書かない**: ローカル絶対パス、個人メールアドレス、ユーザー名。

要件定義書（13/14/15）への参照は文書名で行い、ローカル絶対パスを書かない。

## 8. 受け入れ確認の観点

本書は要件定義のため実行コードを伴わない。以下のランタイム検証は実装フェーズで行う（各 acceptance criteria に対応）。

| 観点 | 期待 | 対応要件 |
|---|---|---|
| 一覧・検索・ページング | 既存一覧契約に沿って絞り込み・ページ送りができ、編集結果が反映される | CMS-1, CMS-2, CMS-3 |
| 新規作成 | コンテンツフィールドのみ入力でき、サーバ管理フィールドは入力欄がない。採番 `id` を確認できる | CMS-5 |
| nested / enum / 配列の編集 | 構造を保って編集でき、enum は許容値選択、配列は項目追加/削除できる | CMS-7, CMS-8, CMS-9 |
| 必須欠落の入力 | 必須欠落が判別され保存が成立しない（サーバ 422 と整合） | CMS-10, CMS-28 |
| 関連 ID 編集 | 実在リソースから選択し、自己参照が抑止される。ダングリングを混入させない | CMS-11, CMS-12 |
| 削除 | 二段階確定で削除でき、被参照時は競合（409）が提示される | CMS-14, CMS-15 |
| 画像アップロード | PNG をアップロードでき、非対応/超過は不適合が伝わる。配信結果が反映される | CMS-17, CMS-18 |
| 認証フロー | admin scope JWT を取得・付与でき、401/403 を区別して扱う。token を露出しない | CMS-22, CMS-23, CMS-24 |
| cross-origin 接続 | 管理面の許可 CORS 範囲で動作する | CMS-25 |
| 編集競合 | 版不一致時に上書きせず再編集に導く（編集消失なし） | CMS-27 |
| エラー提示 | problem+json の検証違反がフィールドに紐づいて可読に表示される | CMS-28 |
| アクセシビリティ | キーボードのみで主要操作が完結し、状態通知が伝わる | CMS-33 |
