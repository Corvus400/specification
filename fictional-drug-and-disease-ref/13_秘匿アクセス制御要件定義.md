# 秘匿アクセス制御要件定義

本書は、架空医薬品・疾患リファレンス backend に CMS（管理画面）および CRUD WebAPI を追加するにあたり、それら管理機能を外部から秘匿しローカル限定とするための**セキュリティ要件 (WHAT)** を定義する。実現手段 (HOW) は本書では規定せず、実装フェーズで別途設計する。

関連文書: CRUD WebAPI の入出力契約は `14_CRUD_WebAPI要件定義.md`、CMS のクライアント側要件は `15_CMS要件定義.md` で定義する。本書はそれらが依拠するアクセス制御の前提を与える。

前提: 本 backend は edge tunnel 経由でのみ外部公開される。既存の読み取り専用 API（`GET /v1/*`）は引き続き外部公開を維持する。本書が対象とするのは、新たに追加される書き込み系 API と管理エンドポイントの**到達制御**である。

---

## 1. 目的

- 管理機能（CMS が操作する CRUD WebAPI および管理エンドポイント）を、外部ネットワークから**到達不能**かつ**存在を秘匿**する。
- 同機能を**ローカル環境からのみ**到達可能とする。
- 既存の公開 API の挙動・公開契約を一切変更しない（回帰禁止）。

---

## 2. 用語・対象スコープ

| 用語 | 定義 |
|---|---|
| 公開面 (public surface) | 外部ネットワークから edge tunnel 経由で到達する経路。既存 `GET /v1/*` を提供する。 |
| 管理面 (admin surface) | CMS／CRUD を提供する、ローカル限定の到達経路。本書の秘匿対象。 |
| 秘匿対象エンドポイント | 下表の path 群。 |

秘匿対象（管理面に置く path）:

| 分類 | path | メソッド |
|---|---|---|
| 医薬品 CRUD（書き込み） | `/v1/drugs`, `/v1/drugs/{id}` | `POST` / `PUT` / `PATCH` / `DELETE` |
| 疾患 CRUD（書き込み） | `/v1/diseases`, `/v1/diseases/{id}` | `POST` / `PUT` / `PATCH` / `DELETE` |
| 画像アップロード | drug 画像アップロード path（`14_CRUD_WebAPI要件定義.md` で定義） | `POST` / `PUT` / `DELETE` |
| 管理エンドポイント | `admin` scope を要する全 path（現行 `/v1/admin/*` を含む） | 全メソッド |

注:

- 既存 `GET /v1/drugs`・`GET /v1/diseases` 等の読み取りは公開面に残す。**書き込みメソッドのみ**を管理面へ分離する。
- `/v1/admin/*`（現行 `whoami` を含む）は現状公開面と同一の到達経路に同居しているが、本要件下ではこれを管理面へ移し外部秘匿の対象とする。本書はこの是正を要件として定める（現状構成の問題点の指摘に留め、解決手段は規定しない）。

---

## 3. 脅威モデルと守るべき性質

外部攻撃者は edge tunnel 経由の公開ドメインにのみ到達できる。守るべき性質:

| ID | 性質 | 内容 |
|---|---|---|
| P1 | 到達不能 | 外部から管理面の path へ到達できない。 |
| P2 | 存在秘匿 | 外部からの管理面 path への試行に対し、エンドポイントの**存在を示唆しない**。「存在するが拒否」（401/403）を外部に返さない。 |
| P3 | ローカル限定 | 管理面はローカル発信のリクエストからのみ到達可能。 |
| P4 | 公開契約秘匿 | 公開される API 契約（OpenAPI）に管理面の path・スキーマが現れない。 |
| P5 | 多層防御 | いずれか単一の防御が破られても秘匿が保たれる。 |

---

## 4. 必須セキュリティ要件（acceptance criteria）

各要件は検証可能な受け入れ基準として記述する。「ローカル専用経路」「公開経路」の具体的な接続先（ポート番号・ホスト名・tunnel 識別子）は本書では規定せず、private な構成情報で SSOT 管理する（§6 参照）。

### 4.1 外部到達不能・存在秘匿

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| SEC-1 | 外部到達不能 | 公開経路（edge tunnel 経由の公開ドメイン）から §2 の管理面 path へリクエストしたとき、当該エンドポイントの処理に到達しない。 |
| SEC-2 | 存在秘匿（404） | 公開経路からの管理面 path への全リクエストは **HTTP 404** を返す。応答に `WWW-Authenticate` ヘッダを含めない。problem+json 本文に、当該エンドポイントの存在・認証要否・拒否理由を示唆する記述（"Forbidden" / "Unauthorized" 等）を含めない。未定義 path に対する通常の 404 と**判別不能**であること。**外部に対して 401／403 を返さない**（「存在するが拒否」を漏らさないため）。 |

### 4.2 ローカル到達後の認証・認可セマンティクス

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| SEC-3 | レイヤ分離 | 到達制御（外部 = 404）と認証・認可（ローカル到達後）は**別レイヤ**とする。**ローカル管理経路に到達したリクエスト**は RFC 準拠の通常セマンティクスに従う（資格情報が無い／不正 → 401、scope 不足 → 403）。この経路に到達できる時点でローカル発信が前提となるため、ローカルでの 401／403 は外部漏洩にならない。SEC-2（外部 404）と本要件（ローカル 401／403）は受信経路で切り分けられ、矛盾しない。 |
| SEC-4 | JWT 認証 | 管理面の全 path は JWT による認証を要する。issuer／audience／署名を検証し、検証失敗は 401（problem+json）。トークン値を応答本文・ログに出さない。 |
| SEC-5 | admin scope 認可 | 管理面の全 path は `admin` scope を要する。scope 不足は 403（problem+json）。401（未認証）・403（認証済み・権限不足）・429（throttle）を取り違えない。 |
| SEC-6 | JWT 取得手段 | CMS／開発者が `admin` scope 付き JWT を**ローカル環境で取得できる手段**を提供する。取得方式の具体は実装フェーズで設計するが、「手段が存在し開発者が token を得られる」ことを要件とする。署名 secret はローカル環境変数等で供給し、ソース・公開ドキュメント・ログに含めない。 |

### 4.3 ローカル限定到達の信頼境界

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| SEC-7 | ローカル限定到達 | 管理面はローカル発信のリクエストからのみ到達可能であること。 |
| SEC-8 | 偽装不可な到達判定 | 到達可否の判定を**クライアントが制御可能な値**（`X-Forwarded-For`／`CF-Connecting-IP` 等の転送ヘッダ、任意ヘッダ）のみに依存させない。**背景**: edge tunnel は origin へ loopback 接続するため、tunnel 経由の外部リクエストもアプリからはローカル送信元に見え得る（既存の rate limit が `CF-Connecting-IP` ヘッダに fallback している事実が傍証）。したがってアプリレベルの loopback CIDR allowlist **単独**ではローカルと外部を区別できず、ローカル限定の担保として不十分である。 |

### 4.4 公開 API 契約面の秘匿

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| SEC-9 | OpenAPI 除外 | 公開される OpenAPI ドキュメントに、§2 の管理面 path・リクエスト／レスポンススキーマが一切含まれないこと。 |

### 4.5 CORS

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| SEC-10 | 管理面 CORS | 管理面は CMS の origin（ローカル開発 origin）に対し `GET/POST/PUT/PATCH/DELETE/OPTIONS` と必要な認証ヘッダ（`Authorization`／`Content-Type`）を許可する。ワイルドカード許可（`anyHost()` 相当）は禁止し、許可 origin を明示列挙する。CMS と管理面は異なる接続先となる **cross-origin 前提**で動作する。 |
| SEC-11 | 公開面 CORS 維持 | 公開面の CORS は現行のまま（`GET` のみ許可、許可 origin は明示列挙、既定無効）を維持し、書き込みメソッドを公開面で許可しない（多層防御の一部）。 |

### 4.6 一貫性

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| SEC-12 | 既存秘匿パターンとの一貫性 | 既存 `/metrics`（edge で 403、アプリで CIDR allowlist による制限）が示す「公開面で管理系を制限する」方針と矛盾しないこと。管理面にはそれより強い「存在秘匿（外部 404）＋到達経路の分離」を適用する。 |

---

## 5. 非機能・運用制約

- **回帰禁止**: 公開面（既存 `GET /v1/*`）の挙動・ステータス・公開契約を変更しない。
- **多層防御**: 到達経路の分離（主防御・外部 404）／JWT + `admin` scope／edge 側の明示 404 ルール／OpenAPI 除外 を重ね、単一層の突破で秘匿が崩れないこと。
- **advisory は制御に数えない**: `robots.txt` 等の advisory はセキュリティ制御として扱わない。秘匿は到達経路の構造的分離と認証で担保する。
- **監査性**: 認証・認可の失敗はログに記録する（トークン・secret 値は記録しない）。

---

## 6. 公開ドキュメントの情報漏洩防止（運用要件）

本書を含む公開ドキュメント（要件定義書・CMS リポジトリの README 等）には、以下を**記載しない**。本要件は 13/14/15 および CMS リポジトリの全公開成果物に適用する。

- **ライブ環境の正確な接続先**: 管理面のポート番号、edge tunnel 識別子、credentials ファイルのパス。
- **秘密情報**: JWT 署名 secret、DB 接続情報・パスワード、各種 API トークン、秘密鍵。
- **個人情報**: ローカル絶対パス、個人メールアドレス、ユーザー名。

これらは private な構成情報（環境変数・非公開設定ファイル）で SSOT 管理し、実装フェーズはそこから実値を解決する。公開ドキュメントは概念レベル（「ローカル専用経路に分離する」等）の記述に留める。

## 7. 受け入れ確認の観点

本書は要件定義のため実行コードを伴わない。以下のランタイム検証は実装フェーズで行う（acceptance criteria に対応）。

| 観点 | 期待 | 対応要件 |
|---|---|---|
| 外部経路から管理面 path へアクセス | 404（`WWW-Authenticate` 無し、未定義 path の 404 と本文同一） | SEC-1, SEC-2 |
| ローカル経路から管理面 path へアクセス | 認証なし = 401 ／ scope 不足 = 403 ／ 正当 = 200 系 | SEC-3, SEC-4, SEC-5 |
| 転送ヘッダ偽装での到達試行 | ヘッダ偽装ではローカル到達できない | SEC-7, SEC-8 |
| 公開 OpenAPI の取得 | 管理面 path・スキーマが含まれない | SEC-9 |
| 公開面 `GET /v1/*` の挙動 | 変更前と不変（回帰確認） | 非機能制約（回帰禁止） |
