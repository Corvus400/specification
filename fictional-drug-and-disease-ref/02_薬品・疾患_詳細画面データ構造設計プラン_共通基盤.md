# 薬品・疾患 詳細画面データ構造設計プラン (共通基盤)

本プランは 3 ファイル構成。共通基盤 (本ファイル) は型規約・件数推奨・API/Fixture 設計差分・参考文献を担う。モデル固有詳細は子ファイルを参照する。

- 医薬品モデル詳細: [03_医薬品モデル詳細.md](03_医薬品モデル詳細.md)
- 疾患モデル詳細: [04_疾患モデル詳細.md](04_疾患モデル詳細.md)

実装真原則: 本プラン記述は `Corvus400/fictional-drug-and-disease-ref-mock-server` の現行実装を 100% 真として整合させる。

<a id="overview"></a>

## 概要

**背景**: [初期素案](01_初期素案_医薬品・疾患・臨床計算ツール統合アプリ.md) は詳細画面のデータスキーマを `id / genericName / brandName / category / dosage / sideEffects[] / contraindications[] / interactions[]` の 7 項目で済ませていた。これでは内服/外用/注射、急性/慢性、規制区分ごとの画面表示分岐、副作用頻度帯ごとの UI 強調、重症度分類 UI の表示制御などが組めない。また素案で指定された「データ件数の目安: 薬品 50 件、疾患 30 件」では、分類軸・UI ブロック発火条件・全列挙値のカバレッジを満たせない。

**目的**: 公開一次情報 ([医療用医薬品 添付文書記載要領 (平成29年/令和5年改正)](https://www.pmda.go.jp/files/000250713.pdf) 26 項目 / [ICD-10 疾病分類 22 章](https://www.mhlw.go.jp/toukei/sippei/) / [ATC分類 5 階層](https://www.kegg.jp/brite/jp08303)) を構造参照し、架空データ用のモデルを「分類軸 × 必須項目」で確定する。データは完全架空、構造のみ一次情報準拠。型化可能な箇所は全て値オブジェクト化し、プリミティブ文字列への雑多押し込めを排除する。

**成果物**: 本プラン 3 ファイル構成で (1) 型規約 (2) 医薬品モデル (3) 疾患モデル (4) 詳細画面 UI パターン (5) 組合せ数計算と推奨データ件数 (6) Fixture/API 設計差分 を集約。

<a id="type-conventions"></a>

## 型規約 (両モデル共通)

サーバ実装は `io.github.corvus400.fictionaldrugdiseaserefmockserver` パッケージ配下で kotlinx.serialization JSON を使用。`plugins/Serialization.kt` で `JsonNamingStrategy.SnakeCase` をグローバル適用し、`encodeDefaults = true` / `ignoreUnknownKeys = true` / `isLenient = true` を設定する。

### プリミティブ

- **文字列**: `String`
- **整数**: `Int`
- **浮動小数**: `Double`
- **真偽値**: `Boolean`
- **リスト**: `List<T>` (`T[]` 表記は本プランで使用しない)
- **Null 許容**: `T?` (trailing `?`)

### 型化ポリシー (最重要)

**自由記述 String を許容するのは「段落型の自由記述が本質で、内部構造が一定しない」場合のみ**。以下 3 条件いずれかを満たすフィールドは値オブジェクト化する:

1. **内部構造が固定**: 「症状 / 処置」「作用機序 / 薬効」「必須項目 / 補助項目」等、サブ見出しが固定。
2. **有限の列挙値**: 頻度帯 (5%以上/1-5%/1%未満/不明)、保管温度 (室温/冷所/冷凍)、腎機能区分 (正常/軽度/中等度/重度/末期) 等。
3. **単位付き数値 / 範囲**: 用量 (mg/g/mL)、年齢範囲 (月齢/歳)、CrCl 範囲 (mL/min)、分子量 等。

自由段落を保つフィールド: `summary`, `etiology`, `DosageInfo.standardDosage`, `Grade.recommendedAction`, `Reference.citation`, `AdverseReaction.symptom`, `PrecautionPopulation.note`, `InteractionEntry.clinicalSymptom`, `InteractionEntry.mechanism`, `Exam.typicalFinding`, `SymptomInfo.mainSymptoms[]` (各要素は短い症状名)。

### Markdown を保持するフィールド

kotlinx.serialization に Markdown 型は存在しないため、**Markdown を保持するフィールドは `String` 型で保持する**。本プラン内でセマンティクス明示のため「**型: `String` / 記法: Markdown**」と併記する。クライアント側で描画する責務:

- **iOS**: `AttributedString(markdown:)` (iOS 15+ 標準、外部依存不要) ([AttributedString documentation](https://developer.apple.com/documentation/foundation/attributedstring))
- **Flutter**: [`flutter_markdown_plus`](https://pub.dev/packages/flutter_markdown_plus) v1.0.7 (Foresight Mobile verified publisher、CommonMark + GFM 対応)。旧 `flutter_markdown` は [2025-05-30 に Google により discontinued](https://github.com/flutter/flutter/issues/162966)。

### Markdown 方言 (両モデル共通)

- **CommonMark + GFM** (表 / 打消し線 / タスクリスト) を許容。両クライアントライブラリがサポート。
- **見出しは `##` `###` のみ**。`#` (h1) は画面タイトルと衝突するため使用しない。
- **画像 `![](...)` は使用しない** (架空データで外部リソースなし)。
- **リンク `[text](url)` は使用しない** (架空データで参照 URL なし)。
- **インライン HTML タグは使用しない**。
- **強調**は `**太字**` のみ使用 (用量の強調など)。`*斜体*` は日本語では視認性が低いため使わない。
- **番号付けは型化後のデータ上では保持しない** (番号付き段落は `List<NumberedParagraph>` の `order: Int` で構造化する。クライアント側で `N.M.` 形式にレンダリングする)。

### 日付・日時

Flutter `DateTime.parse`、Swift `ISO8601DateFormatter`、Kotlin `LocalDate.parse` が標準でパース可能な **ISO 8601** を採用する。粒度別に以下 2 種に限定する:

| 用途 | 形式 | 例 | 理由 |
|------|------|-----|------|
| 添付文書改訂年月 (`revisedAt`) | `String` / `YYYY-MM-DD` | `"2024-03-01"` | 添付文書記載要領は「年月」精度。日は `01` 固定で時刻不要、タイムゾーン問題を完全回避 |
| 閲覧履歴・ブックマーク等の操作日時 | `String` / `YYYY-MM-DDTHH:mm:ss` | `"2026-04-20T09:30:00"` | クライアント永続化スキーマの参照型 |

### 列挙型 (Enum)

kotlinx.serialization は `@Serializable enum class` を第一級サポート ([kotlinx.serialization docs — Customize Enum Entry Serial Names](https://github.com/Kotlin/kotlinx.serialization/blob/master/docs/builtin-classes.md))。以下方針:

- **フィールド型に enum class を直接使う** (String への降格はしない)。型安全性をサーバ側と Fixture 検証で担保する。
- Kotlin の enum constant は英語識別子 (例: `TABLET`) とし、JSON ワイヤ値は **英語 snake_case** を `@SerialName("tablet")` で固定する。旧日本語ラベル (例: `錠剤`) は entry 上部の `/** 錠剤 */` KDoc に保存し、Kotlin source / IDE hover 上から即座に和訳を確認できる状態にする。
  - (a) サーバ実装は型安全 (タイポはコンパイルエラー)
  - (b) Kotlin source / IDE hover 上の人間可読性: 旧日本語ラベルが KDoc に残るため、英語 SerialName を読みながら和訳を即時確認できる
  - (c) URL クエリ ASCII 化: `?regulatory_class=prescription_required` のように URL エンコード不要で透過的に扱える
  - (d) grep 容易性: 全 enum 値が `[a-z][a-z0-9_]*` 形式なので source / log / fixture を横串 grep できる
  - (e) i18n 拡張容易性: JSON ワイヤ値と表示ラベルが分離されたため、クライアント側で多言語ラベル辞書を持つだけで多言語化できる
  - クライアント (Flutter freezed / Swift) は JSON 上の英語文字列として受信し、各言語のクライアント側 enum にマップできる
- **全 `@Serializable enum class` に `val serialName: String` extension property を必須実装**: `serializer().descriptor.getElementName(index = ordinal)` で descriptor 経由参照する SSOT property。production / fixture / OpenAPI description から SerialName literal を直接書かず、必ずこの property 経由で参照する。
- 複数値を取る場合は `List<EnumType>`。
- enum constant に追加属性を JSON に出したい場合は **enum ではなく `data class` で表現** する (kotlinx.serialization は enum を必ず文字列 1 個で表現するため)。

具体例:
```kotlin
@Serializable
enum class DosageForm {
    /** 錠剤 */
    @SerialName("tablet") TABLET,
    /** カプセル */
    @SerialName("capsule") CAPSULE,
    /** 注射剤 */
    @SerialName("injection_form") INJECTION_FORM,
    // ...
    ;

    val serialName: String
        get() = serializer().descriptor.getElementName(index = ordinal)
}
// Drug.dosageForm: DosageForm → JSON: "dosage_form": "tablet"
// IDE hover で `TABLET` を見ると KDoc 「錠剤」が表示される
```

### 空配列・null ポリシー

- 任意フィールドは **`T? = null` か `List<T> = emptyList()` の二択**。両立 (`List<T>?`) は禁止。
- リスト型は null を使わず `List<T> = emptyList()` に統一。`encodeDefaults = true` により空配列は JSON 上 `[]` として出力される。クライアント (Flutter freezed / Swift Codable) は `null` と空配列の両方を許容する実装で互換性を保つ。
- 値オブジェクト型 (`data class`) で「有無」を表すフィールドは `T? = null` を使う (例: `overdose: OverdoseInfo? = null`)。

### `@Serializable data class` 階層の指針

- 複数フィールドが一単位として意味を持つ場合は必ず `data class` に切り出す (例: `Dose(amount, unit)`、`AgeRange(min, max, label)`、`PharmacokineticsInfo(...)`)。
- `data class` を `@Serializable` にすると、kotlinx.serialization が nested 構造を自動で階層 JSON に展開 ([kotlinx.serialization docs — nested serializable objects](https://github.com/Kotlin/kotlinx.serialization/blob/master/docs/basic-serialization.md))。

## API / Fixture 設計差分

### ベースパス

全データ系エンドポイントは **`/v1/`** プレフィックス配下。`/v1` を除いた絶対パスは存在しない (Admin API `/__admin/` および `/health` は別系統)。

### エンドポイント一覧

| メソッド | パス | 用途 |
|---------|-----|-----|
| GET | `/v1/drugs` | 医薬品一覧 (フィルタ + ページング) |
| GET | `/v1/drugs/{id}` | 医薬品詳細 |
| GET | `/v1/diseases` | 疾患一覧 (フィルタ + ページング) |
| GET | `/v1/diseases/{id}` | 疾患詳細 |
| GET | `/v1/categories` | フィルタ選択肢一括取得 |
| GET | `/v1/images/dosage-forms/{form}` | 剤形画像 (PNG バイナリ) |
| GET | `/v1/images/drugs/{drugId}` | 医薬品個別画像 (PNG バイナリ、特定 ID のみ) |

詳細仕様は [`#image-endpoints`](#image-endpoints) と各エンドポイント節を参照。

### レスポンス形式

- 一覧系 (`/v1/drugs` / `/v1/diseases`): 共通フィールドのうち「リスト画面に必要な最小項目」のみ返す。JSON キーは snake_case (kotlinx.serialization の `JsonNamingStrategy.SnakeCase` で統一変換)。各言語のモデル識別子は言語 idiom (Kotlin/Dart/Swift は camelCase) を維持。
  - **DrugSummary 出力フィールド** (10 個): `id / brand_name / generic_name / therapeutic_category_name / regulatory_class / dosage_form / brand_name_kana / atc_code / revised_at / image_url`
  - **DiseaseSummary 出力フィールド** (8 個): `id / name / icd10_chapter / medical_department / chronicity / infectious / name_kana / revised_at`
- 詳細系 (`/v1/drugs/{id}` / `/v1/diseases/{id}`): 全共通フィールド + 入れ子構造を全展開で返す。`disclaimer` フィールドが root に必須付与される (詳細は [`#disclaimer-field`](#disclaimer-field))。
- 空値ポリシー: `Serialization.kt` が `encodeDefaults = true` / `ignoreUnknownKeys = true` で設定済。`List<T> = emptyList()` は JSON 上 `[]`、`T? = null` は JSON 上 `null` として出力される。クライアント (Flutter freezed / Swift Codable) は `null` と空配列の両方を許容する実装で互換性を保つ。

### ページネーション envelope

一覧系レスポンスの wrapper は `DrugListResponse` / `DiseaseListResponse` data class で表現され、JSON では以下 6 フィールドを root に持つ:

```
{
  "items": [ ... ],
  "page": 1,
  "page_size": 20,
  "total_pages": 6,
  "total_count": 120,
  "disclaimer": "FICTIONAL DATA - NOT FOR MEDICAL USE / 架空データ・医療判断には使用不可"
}
```

- `page` / `page_size` / `total_pages` / `total_count` は `Int`。
- `disclaimer` は `Disclaimer.SHORT` 定数 (`config/Disclaimer.kt`) のデフォルト値。一覧 envelope にも root に必ず付く。
- クライアント側変数命名は idiom 維持で camelCase (例: Kotlin `pageSize`、Dart `pageSize`、Swift `pageSize`)。HTTP/JSON 側は snake_case で固定。

<a id="disclaimer-field"></a>

### `disclaimer` フィールド

以下 4 レスポンス root に `disclaimer: String` を必須付与する (デフォルト値 `Disclaimer.SHORT`):

- `Drug` (詳細)
- `DrugListResponse` (一覧 envelope)
- `Disease` (詳細)
- `DiseaseListResponse` (一覧 envelope)

加えて `ErrorResponse` (エラーボディ) の root にも `disclaimer: String` が必須付与される。

`CategoriesResponse` には `disclaimer` を付けない (規約: フィルタ選択肢取得は医療判断に直接関与しない)。

`Disclaimer.SHORT` 値: `"FICTIONAL DATA - NOT FOR MEDICAL USE / 架空データ・医療判断には使用不可"`

`Disclaimer.FULL_JA_EN` 値 (起動時ログ用): 5 行構成の日本語+英語免責文 (`config/Disclaimer.kt`)。レスポンスには使用しない。

<a id="categories-response"></a>

### `/v1/categories` 返却構造

```
{
  "atc": [{ "code": "A", "label": "消化器系および代謝" }, ...],
  "therapeutic_categories": [{ "id": "alimentary_metabolism", "label": "消化器系および代謝" }, ...],
  "route_of_administration": ["oral", "topical", "injection_route", ...],
  "dosage_form": ["tablet", "capsule", "injection_form", ...],
  "regulatory_class": ["poison", "potent", "ordinary", ...],
  "icd10_chapters": [{ "roman": "I", "code": "A00-B99", "label": "感染症および寄生虫症" }, ...],
  "medical_departments": ["internal_medicine", "cardiology", "infectious_disease", ...]
}
```

ラベル軸の方針:

- `atc[].label` / `therapeutic_categories[].label` / `icd10_chapters[].label` は **日本語維持**。これらは enum SerialName とは独立した日本語マスタ (ATC 第一階層名 / 治療分類名 / ICD10 章日本語訳) を表示するため、英語 snake_case の SerialName 化対象ではない。
- `therapeutic_categories[].id` は `TherapeuticCategory` の `categoryId` 属性 (例: `alimentary_metabolism`) であり、enum constant 名 (例: `ALIMENTARY_METABOLISM`) ではない。`/v1/drugs?therapeutic_category=...` クエリは **enum constant 名** を要求するため、クライアント側で id↔constant 名のマッピングは行わず、`/v1/categories` の `id` を表示用 key として保持しつつ、絞り込みクエリ送信時はクライアントの enum constant 名に変換する。
- enum 由来 4 フィールド (`route_of_administration` / `dosage_form` / `regulatory_class` / `medical_departments`) は **英語 SerialName のみの `List<String>`**。クライアントは英語 SerialName を受信した上で、自身の多言語ラベル辞書から表示文言を解決する。
- `disclaimer` フィールドは `CategoriesResponse` には含まれない (上記 [`#disclaimer-field`](#disclaimer-field) 参照)。

UI のフィルターピッカーがこのエンドポイント 1 回で全選択肢を取得できる。`CategoriesResponse` は起動時に固定 drugs (120 件) と enum 宣言から一度だけ導出され、リクエストごとに常に同じ内容を返す (シナリオ切替の対象外)。

### エラーレスポンス

```
{
  "code": String,
  "message": String,
  "details": String?,
  "disclaimer": String
}
```

HTTP 4xx/5xx でも body は JSON。`code` は以下の値を取り得る:

- `BAD_REQUEST` — 必須パスパラメータ欠落等
- `NOT_FOUND` — 詳細取得で id が存在しない
- `INVALID_THERAPEUTIC_CATEGORY` — `therapeutic_category` クエリの値が `TherapeuticCategory` の constant 名に該当しない
- `INVALID_PRECAUTION_CATEGORY` — `precaution_category` クエリの値が `PrecautionPopulationCategory` の constant 名に該当しない
- `INVALID_ONSET_PATTERN` — `onset_pattern` クエリの値が `OnsetPattern` の constant 名に該当しない
- `INVALID_EXAM_CATEGORY` — `exam_category` クエリの値が `ExamCategory` の constant 名に該当しない
- `INVALID_SORT_KEY` — `sort` クエリの値が許容ソートキーに該当しない
- `INTERNAL_ERROR` — 5xx 系の汎用エラー

`details` は型 `String?`。`disclaimer` は `Disclaimer.SHORT` がデフォルト付与される。

<a id="image-endpoints"></a>

## 画像エンドポイント

両エンドポイントはバイナリ (`image/png`) を返す。

### `GET /v1/images/dosage-forms/{form}`

| 項目 | 仕様 |
|-----|-----|
| パスパラメータ | `form` — `DosageForm` enum SerialName (例: `tablet`、`capsule`、`injection_form`) |
| クエリ `size` | `S` \| `M` \| `Original` (省略時 `Original`)。それ以外は HTTP 400 |
| 200 | `image/png` バイナリ |
| 404 | 該当 form のリソースが存在しない場合 |
| サポート 13 form | `tablet` / `capsule` / `powder` / `granule` / `liquid` / `injection_form` / `ointment` / `cream` / `patch` / `eye_drops` / `suppository` / `inhaler` / `nasal_spray` |

### `GET /v1/images/drugs/{drugId}`

| 項目 | 仕様 |
|-----|-----|
| パスパラメータ | `drugId` — `drug_NNNN` 形式 |
| クエリ `size` | `S` \| `M` \| `Original` (省略時 `Original`) |
| 200 | `image/png` バイナリ (`drug_0080`、`drug_0089` の 2 件のみ実画像が存在する) |
| 404 | 上記 2 ID 以外、または size リソース欠落時 |

### `image_url` フィールドの生成ロジック

`Drug` 詳細および `DrugSummary` 一覧の root に必須付与される `image_url` は、以下の分岐で生成される (`model/drug/DrugImageUrlBuilder.kt`):

```
if drug.id in {"drug_0080", "drug_0089"}:
    image_url = "/v1/images/drugs/{drug.id}?size=Original"
else:
    image_url = "/v1/images/dosage-forms/{drug.dosageForm.serialName}?size=Original"
```

クライアントは受信した `image_url` を base URL と結合してそのまま GET する。クライアント側で剤形マッピングを再実装する必要はない。

<a id="fixture-soft-constraints"></a>

## 仕様制約一覧

サーバ起動時に `DrugFixtureValidator` / `DiseaseFixtureValidator` / `CrossReferenceValidator` が以下の制約を検証する。違反があればサーバは起動失敗する。

### 医薬品データ整合性

| 制約 | 条件 |
|------|------|
| `Drug.contraindications` ≥1 | 全件 |
| `Drug.indications` ≥1 | 全件 (例外: `drug_0080` は治療目的を持たない fixture として空を許容) |
| `Drug.packages` ≥1 | 全件 |
| `Drug.regulatoryClass` ≥1 | 全件 (普通薬は `ORDINARY` を明示) |
| `Drug.pharmacokinetics` 非 null | 注射剤 (`dosageForm = INJECTION_FORM` または `routeOfAdministration = INJECTION_ROUTE`) |
| `Drug.pharmacokinetics.bloodConcentration` 非 null | 注射剤 |
| `Drug.pharmacokinetics.metabolism` または `.excretion` のいずれか非 null | 注射剤 |
| `Drug.administrationPrecautions` ≥1 | 注射剤 + 外用剤 (`dosageForm ∈ {OINTMENT, CREAM, PATCH, EYE_DROPS, NASAL_SPRAY, SUPPOSITORY}`) |
| `Drug.warning` ≥1 | 毒薬・劇薬 (`regulatoryClass` に `POISON`/`POTENT` 含む) または生物由来製品 (`BIOLOGICAL`/`SPECIFIED_BIOLOGICAL` 含む) |
| `Drug.handlingPrecautions` ≥1 | 生物由来製品 |
| `Drug.insuranceNotes` ≥1 | 麻薬・向精神薬 (`regulatoryClass` に `NARCOTIC`/`PSYCHOTROPIC_*` 含む) |
| `Drug.dosageRelatedPrecautions` ≥1 | 慢性疾患長期服用薬 (`atcCode` 第 1 文字が `A` または `C`) |

ID 整合性: `drug_NNNN` 4 桁ゼロ埋め一意 + 連番 (`drug_0080` 等のオーバーライド ID は連番外を許容)。

### 疾患データ整合性

| 制約 | 条件 |
|------|------|
| `Disease.symptoms.mainSymptoms` ≥1 | 全件 |
| `Disease.requiredExams` ≥1 | 全件 |
| `Disease.medicalDepartment` ≥1 | 全件 |
| `Disease.diagnosticCriteria.required` ≥1 | 全件 |
| `Disease.severityGrading.grades` ≥2 | `severityGrading` 非 null 時 |
| `Disease.infectious = true` | `icd10Chapter = CHAPTER_I` |
| `Disease.epidemiology` 非 null | `CHAPTER_I` / `CHAPTER_XV` |
| `Disease.epidemiology.riskFactors` ≥1 | `CHAPTER_I` |
| `Disease.epidemiology.riskFactors` に感染経路キーワードを含む | `CHAPTER_I` |
| `Disease.symptoms.onsetPattern` 非 null | `CHAPTER_I` |
| `Disease.severityGrading` 非 null | `CHAPTER_II` / `CHAPTER_IX` |
| `Disease.prognosis` 非 null かつ非空 | `CHAPTER_II` |
| `Disease.treatments.pharmacological` ≥1 | `CHAPTER_IV` |
| `Disease.symptoms.mainSymptoms` ≥3 | `CHAPTER_V` |
| `Disease.requiredExams` 内に `ExamCategory.IMAGING` を最低 1 含む | `CHAPTER_IX` |
| `Disease.treatments.acutePhaseProtocol` ≥1 | `CHAPTER_IX` かつ `chronicity = ACUTE` |
| `Disease.epidemiology.onsetAgeRange` 非 null | `CHAPTER_XV` |
| `Disease.epidemiology.sexRatio` 非 null | `CHAPTER_XV` |
| `Disease.prevention` ≥1 | `infectious = true` または `EpidemiologyInfo.riskFactors` 非空 |

ID 整合性: `disease_NNNN` 4 桁ゼロ埋め一意 + 0 から `件数 - 1` まで連番 (穴なし)。

### 横断参照整合性

| 制約 | 条件 |
|------|------|
| `relatedDrugIds` / `relatedDiseaseIds` / `InteractionEntry.drugId` / `PharmaTreatment.drugIds` の参照先 ID は実在する fixture を指す | ダングリング参照禁止 |
| `relatedDrugIds` に向精神薬 (`regulatoryClass` に `PSYCHOTROPIC_1` / `PSYCHOTROPIC_2` / `PSYCHOTROPIC_3` のいずれかを含む薬品) を最低 1 件含む | `icd10Chapter = CHAPTER_V` |
| `relatedDrugIds` に妊婦禁忌薬 (`precautionsForSpecificPopulations[].category` に `PREGNANT` を含む薬品) を含めない | `icd10Chapter = CHAPTER_XV` |

<a id="fixture-guidance"></a>

## Fixture 作成ガイダンス

- 架空名は素案規約どおり ("サンプルチン" 等、実在名排除)。
- 薬品 120 件の分布は [03_医薬品モデル詳細.md `#combination-counts`](03_医薬品モデル詳細.md#combination-counts) 「分布設計」に従う。詳細画面の全 UI パターンが各最低 1 件で発火するよう分散配置。
- 疾患 80 件の ICD-10 章分布は [04_疾患モデル詳細.md `#combination-counts`](04_疾患モデル詳細.md#combination-counts) 「分布設計」に従う。主要 6 章 (I/II/IV/V/IX/XI) に重点配置し、全 22 章で最低 1 件カバー。
- `relatedDrugIds` / `relatedDiseaseIds` / `InteractionEntry.drugId` / `PharmaTreatment.drugIds` は必ず実在 id を参照 (ダングリング参照禁止)。
- 日付フォーマットは ISO 8601 で `IsoDateFormatter` 経由で生成 (`.claude/rules/fixture-quality.md` 第 1 項)。
