# 閲覧履歴 DB スキーマ定義

親プラン: [05_医薬品・疾患マスタアプリ詳細化プラン.md][plan] (閲覧履歴画面、永続化設計)

## 目的

ユーザーが詳細画面で閲覧した医薬品・疾患の ID を自動記録し、直近の閲覧から短時間で再アクセスできるようにする。医療ツール想定で「直前に見た薬剤に戻る」頻度が高いため、検索・ブックマークと独立した導線として提供する。

## テーブル定義

### `browsing_history` (Flutter drift / iOS SwiftData 共通スキーマ)

| カラム | 型 | 制約 | 説明 |
|-------|----|------|------|
| `id` | String | PRIMARY KEY | 閲覧対象 ID (`drug_NNNN` または `disease_NNNN`) |
| `viewedAt` | DateTime | NOT NULL | 最終閲覧日時 (ISO 8601 UTC、同一エンティティ再閲覧時に上書き) |

`id` は `drug_NNNN` / `disease_NNNN` 形式で名前空間分離されているため `id` 単独で一意。種別判定は `id` 接頭辞 (`drug_` / `disease_`) で行い、専用カラムは持たない。`snapshot` は保存しない (ブックマークと異なり、オフライン閲覧保証の対象外)。

### インデックス

- `idx_browsing_date ON browsing_history(viewedAt DESC)` — 時系列ソート
- `idx_browsing_id_prefix_date ON browsing_history(substr(id, 1, 5), viewedAt DESC)` — タブ切替時のフィルタ (`drug_` / `disease_` の先頭 5 文字フィルタ)

タブ別フィルタの SQL 例: `SELECT * FROM browsing_history WHERE id LIKE 'drug_%' ORDER BY viewedAt DESC`。

## 使用箇所

### 画面配置: 独立画面 `/history` (閲覧履歴画面)

検索画面・ブックマーク画面と並列の独立エントリ。

- ルート (Flutter): `/history` (go_router)
- View (iOS): `BrowsingHistoryView`
- UI: 3 タブ切替 (すべて / 医薬品 / 疾患) + 時系列降順リスト + 削除 + 全削除アクション
- リスト項目: `id` から当該エンティティ名を表示するため、ブックマーク `bookmarks.snapshot` を兼用参照する (ブックマーク済の場合)。未ブックマークの履歴項目は API 取得 (`/v1/drugs/{id}` / `/v1/diseases/{id}`) で表示用最小項目を取得しメモリキャッシュする
- タップ挙動: 該当詳細画面へ遷移 (オンライン取得、失敗時はエラー表示)

検索画面内のサブセクション案・ブックマーク画面のタブ統合案は採用せず: 閲覧履歴 (自動記録) とブックマーク (明示保存) は意味論が異なるため UI 統合しない。

## ライフサイクル

| 操作 | トリガ | DB 操作 |
|-----|-------|--------|
| 作成 / 更新 | 詳細画面 (`DrugDetailView` / `DiseaseDetailView`) 遷移時 | `UPSERT` で `viewedAt` を現在時刻に更新。同一 `id` は上書き |
| 削除 (単件) | 閲覧履歴画面でスワイプ削除 | `DELETE FROM browsing_history WHERE id = ?` |
| 削除 (全件) | 閲覧履歴画面の全削除ボタン | `DELETE FROM browsing_history` |
| 自動削除 | 件数上限超過時 | 最古の `viewedAt` から順に削除 |

## 件数上限

上限 100 件 (FIFO 自動削除)。UPSERT のため同一エンティティ再閲覧では件数は増えない。

## 採用理由

独立画面 `/history` を採用。代替案: ブックマーク画面に 4 タブ目として追加は自動記録と明示保存の意味論差から却下、検索画面ホームの「最近見た」セクションは一覧性低く全履歴確認には別画面が必要なため却下、画面なしは永続化する意味がないため却下。
