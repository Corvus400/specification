## アプリ仕様

### コンセプト

医薬品・疾患・臨床計算ツールを統合した医療従事者向けリファレンス。データは完全架空だが、実運用アプリと同等の UX / アーキテクチャ深度を提供する。

### 画面構成 (iOS / Flutter 共通 4 画面 + 計算ツール)

| # | 画面 | 主要要素 | アーキテクチャ 4 要素 |
|---|------|---------|---------------------|
| 1 | 検索一覧 (ホーム兼用) | 検索バー + カテゴリフィルター + ページングリスト + 最近の検索履歴 | Navigation + 非同期 (API) + 永続化 (履歴) |
| 2 | 詳細 | 一般名/商品名/薬効分類/用法用量/副作用/禁忌/相互作用 (マークダウン描画) + ブックマークトグル | 状態管理 + 永続化 (ブックマーク) |
| 3 | ブックマーク一覧 | 保存済み薬品/疾患リスト + 削除 | 永続化 |
| 4 | 計算ツール | BMI / eGFR / CrCl のフォーム入力 → 即時結果 + 計算履歴 | 状態管理 (Form) + 永続化 (履歴) |

4 要素 (Navigation / 状態 / 非同期 / 永続化) が全画面で網羅される ([Apple Scrumdinger Tutorial](https://developer.apple.com/tutorials/app-dev-training/) / [Flutter Compass Architecture Case Study](https://docs.flutter.dev/app-architecture/case-study))。

### 特徴機能

- **オフライン対応**: 初回起動後は全データローカルキャッシュで動作
- **全文検索**: iOS は SwiftData + `@Query` 述語、Flutter は drift + FTS5
- **マークダウン描画**: 詳細画面の本文は Markdown でレンダリング (iOS は AttributedString / Flutter は `flutter_markdown`)
- **臨床計算ツール**: BMI (体重/身長²)、eGFR (日本人係数式)、CrCl (Cockcroft-Gault 式)。式の出典は README に明記
- **ダークモード**: SwiftUI `@Environment(\.colorScheme)` / Flutter `MediaQuery.platformBrightness` 自動追従
- **iPad Universal**: SwiftUI `NavigationSplitView` / Flutter `LayoutBuilder` + 600px 分岐

## データ設計 (架空薬品方針)

### 必須遵守事項

- 実在薬品名・成分名・商品名は一切使用しない
- 架空名は明らかに架空と判別できる命名 (例: 「サンプルチン」「テスタマイン」「モックロン」)
- 薬効分類名は一般カテゴリ (解熱鎮痛薬 / 抗生物質 / 糖尿病薬 等) を使用可
- README 冒頭に免責文言を明記: 「このアプリのデータは架空薬品のサンプルであり、医療判断・服薬判断には一切用いないでください」
- データ件数の目安: 薬品 50 件、疾患 30 件、各カテゴリに分散

### スキーマ (ktor-mock-server レスポンス / 両クライアント共通)

PMDA 添付文書項目を構造参照したフィールド構成:

```json
{
  "id": "drug_0001",
  "genericName": "サンプルチン",
  "brandName": "サンプルチン錠 100mg",
  "category": "解熱鎮痛薬",
  "dosage": "成人1回100mg、1日3回食後経口投与",
  "sideEffects": ["悪心", "発疹", "頭痛"],
  "contraindications": ["本剤の成分に対し過敏症の既往歴のある患者"],
  "interactions": [
    {"with": "テスタマイン", "note": "併用により血中濃度上昇の可能性"}
  ]
}
```

疾患エンティティも同様に `id / name / category / symptoms / treatments / relatedDrugIds` を定義する。

## API / 通信設計

### ktor-mock-server 責務

- 薬品/疾患の固定 JSON を提供する REST API サーバー
- Oisix 既存資産の型安全モックサーバー設計を流用
- API キー不要、ローカル起動のみ、リポジトリ同梱

### エンドポイント

| メソッド | パス | 用途 |
|---------|------|------|
| GET | `/drugs?q={query}&category={cat}&page={n}` | 薬品検索 (ページング付き) |
| GET | `/drugs/{id}` | 薬品詳細 |
| GET | `/diseases?q={query}&category={cat}&page={n}` | 疾患検索 |
| GET | `/diseases/{id}` | 疾患詳細 |
| GET | `/categories` | カテゴリ一覧 |

### プロトコル選定

REST を採用。GraphQL は自前定義のためどちらも選択可能だが、オーバーフェッチ問題が発生しないデータ規模で、Oisix REST 実績をそのまま転用できるメリットが大きい。

## 永続化設計

### 保存対象 (最小限ポリシー)

- **ブックマーク**: `id / type (drug|disease) / bookmarkedAt` の 3 カラム 1 テーブル

### マイグレーション戦略

- iOS: `VersionedSchema` + `SchemaMigrationPlan` ([Versioning Your SwiftData Schema — Michael Tsai 2026-02-18](https://mjtsai.com/blog/2026/02/18/versioning-your-swiftdata-schema/))
- Flutter: drift `schemaVersion` + `MigrationStrategy`
- 初期 v1 のみ。v1→v2 マイグレーションサンプルを 1 つ実装して README に記載 (ポートフォリオ訴求)
- CloudKit は不採用 ([SwiftData+CloudKit Migration Failing — Apple Developer Forums](https://developer.apple.com/forums/thread/742899) の既知バグ回避)

#### 完了確認
```bash
# サーバー起動後、全エンドポイントが 200 と妥当な JSON を返す
./gradlew run &
curl -s http://localhost:8080/drugs?q=サンプル | jq '.[] | .genericName' | head -5
curl -s http://localhost:8080/drugs/drug_0001 | jq '.category'
curl -s http://localhost:8080/categories | jq 'length'
```

#### コミット

## リスクと対策

| リスク | 対策 |
|--------|------|
| SwiftUI + TCA 同時未経験による工数肥大 | TCA Hello World ゲート (XS) で早期判定、不能なら `@Observable` + MVVM にダウングレード |
| 架空データに実在薬品名が紛れる薬機法リスク | Fixture に grep チェックを CI 組み込み、実在名 0 件を CI 成功条件とする |
| iPhone 実機がなく iPad のみで検証する UX の不整合 | iPad の `NavigationSplitView` 対応を最初から入れ、iPhone サイズはシミュレータで動作確認 |
| Flutter 公式 MVVM を選ばない判断へのレビュアー疑問 | README に「訴求先 (UPSIDER / LayerX / スタディプラス) の採用パターンに合わせて Clean + Contract Pattern を選定」と明示 |
| iOS 26 Liquid Glass への追従不足 | SwiftUI 標準コンポーネントで揃え、iOS 26+ 限定機能は `#available(iOS 26, *)` で段階採用 |
| 題材 A (医療) と iOS 訴求先 (非医療) のジャンル不一致 | README に「ドメインではなく技術訴求として作成。同一題材を TCA / Clean + Contract で実装した比較」と明示 |
| CloudKit バグでのデータ破損 | CloudKit 不採用。ローカルのみで完結 |
