# 医薬品・疾患マスタアプリ詳細化プラン (素案3-3 の要件確定版)

## 概要

[初期素案][sketch] で定義された「医薬品・疾患・臨床計算ツールを統合した医療従事者向けリファレンスアプリ」の要件を、以下の観点で具体化する。

- 画面仕様 (検索 / 医薬品詳細 / 疾患詳細 / ブックマーク / 閲覧履歴 / 計算ツール、計 6 画面) の責務と相互関係
- 検索画面の機能要件 (キーワード欄・絞り込み軸・ソート軸)
- Flutter / iOS 技術スタック (素案からの欠落項目を補完し、対称形で整理。未選定項目は明示)
- 運用・非機能要件 (永続化設計 / ページネーション / ローディング / エラー・Empty)

モデル・型のデータ構造は [02_薬品・疾患_詳細画面データ構造設計プラン_共通基盤.md][base] (共通基盤)、[03_医薬品モデル詳細.md][drug] (医薬品)、[04_疾患モデル詳細.md][disease] (疾患) に委譲する。永続化対象の DB スキーマ (ブックマーク / 閲覧履歴 / 計算履歴 / 検索履歴 / テーマ設定) は [bookmark][bm] / [browsing-history][bh] / [calculation-history][ch] / [search-history][sh] / [theme-settings][th] に委譲する。本プランではリンク参照のみで本文を複製しない。

実装真原則: 本プラン記述は `Corvus400/fictional-drug-and-disease-ref-mock-server` の現行実装を 100% 真として整合させる。アプリ実装 (Flutter / iOS) は別セッションで行う。

ポートフォリオ目的 (HOKUTO/メドレー型企業への就職活動素材) で、データは完全架空。対象 OS: iOS (SwiftUI) / Flutter の並行実装。

## 基本方針

- **データ層統合・UI 層分離**: ブックマーク・閲覧履歴は `id` 接頭辞 (`drug_` / `disease_`) で統合 1 テーブル管理 ([bookmark][bm] / [browsing-history][bh])、API は `/v1/drugs` / `/v1/diseases` で分離 ([base #api-spec][base-api])、UI は統合/分離を画面目的ごとに選ぶ
- **医薬品と疾患は独立ドメイン**: 詳細画面は完全分離、相互参照はナビゲーション遷移でつなぐ (1 画面同時表示はしない)
- **計算ツールは独立モジュール**: 医薬品/疾患/ブックマークと結合しない
- **永続化は 5 種類を別ファイルに分離**: ブックマーク / 閲覧履歴 / 計算履歴 / 検索履歴 / テーマ設定。スキーマ・ライフサイクル・画面配置は「運用・非機能要件」→「永続化設計」 + 各別ファイル参照
- **ブックマークはスナップショット戦略**: API リスト最小項目 (DrugSummary 10 / DiseaseSummary 8) + `bookmarkedAt` を保存 (詳細は [bookmark][bm])
- **OSS は Deprecated 排除**: 「調査結果サマリ」→「OSS 差し替え」参照
- **実装品質方針**:
  - **SSOT 遵守**: URL / ベース URL / ポート番号 / タイムアウト値 / ページングサイズ / 件数上限 / 配色などのマジックナンバーは定数・config・環境変数に一元化する
  - **エラー可視性**: catch / `do-catch` で捕捉したエラーは `OSLog` / `dart:developer.log` 等で必ずログ出力するか上位レイヤに伝播する。空 catch・サイレント握りつぶしは禁止。UIState は Error / Empty を明示的な case として表現

## 調査結果サマリ

### 既定事項

- 画面構成: 素案 4 画面 + 計算ツール + 閲覧履歴 = 計 6 画面
- API エンドポイント: `/v1/drugs` / `/v1/diseases` 分離 + `/v1/categories` + 画像 2 件 = 計 7 件 ([base #api-spec][base-api])
- 永続化方針: ブックマーク・閲覧履歴は `id` 接頭辞で drug/disease 統合、計算履歴は独立 ([素案 §永続化][sketch])
- 計算ツール: 独立画面、医薬品/疾患データとの連動なし。式は BMI = 体重/身長²、eGFR = 日本人係数式、CrCl = Cockcroft-Gault ([素案][sketch] L90、具体式は実装側 README に明記)
- 架空データ件数: 医薬品 120 件 / 疾患 80 件 / 合計 200 件 ([base #count-recommendation][base-count])
- 型規約 / Markdown 方言 / 空配列ポリシー / 日付日時形式: [base #type-conventions][base-types] に一元管理
- ページング既定値: `page_size = 20` / 上限 100 (mock-server `SearchDefaults.DEFAULT_PAGE_SIZE` / `MAX_PAGE_SIZE` と整合)
- envelope に `has_more` フラグはない。最終ページ判定はクライアント側で `page >= total_pages` または `items.length < page_size` で行う

### OSS 差し替え (Deprecated 排除)

素案の Flutter Markdown OSS `flutter_markdown` は Google による 2025-05-30 discontinued ([flutter/flutter#162966][fm-deprecate])。確定採用版は [flutter_markdown_plus v1.0.7][fmplus]。iOS 側は [AttributedString(markdown:)][attrstr] (iOS 15+ 標準) 採用で Deprecated 問題なし。

### 医薬品×疾患の詳細画面構成

- ブロック数: 医薬品 D1-D19 ([drug #ui-patterns][drug-ui]) / 疾患 E1-E18 ([disease #ui-patterns][disease-ui])
- 共通ブロック: ヘッダー / 分類バッジ / クロスリンク列 / ブックマークトグル固定フッター / 改訂日
- 医薬品固有: 画像 (image_url) / 警告バナー / 用法 4 タブ (標準・小児・腎・肝) / 相互作用 2 タブ / 副作用 / 薬物動態折り畳み
- 疾患固有: 概説 / 疫学 / 原因病態 / 症状 / 診断基準 / 検査 / 重症度分類 / 鑑別合併症 / 治療 3 タブ / 予後 / 予防
- 相互リンク: 医薬品→疾患 (`relatedDiseaseIds`, [drug D18][drug-ui]) / 疾患→医薬品 (`relatedDrugIds`, [disease E15][disease-ui]) / 疾患→疾患 ([disease E16][disease-ui]) / 疾患の治療ブロック内 `drugIds` ([disease #disease-model][disease-model])。1 画面同時表示なし (カルーセル→別画面遷移)

### エンティティサイズ推定 (ブックマーク戦略の根拠)

| モデル | 全フィールド数 | 最大ネスト | Markdown 長文推定 | 1 件推定 JSON サイズ | 全件合計 |
|--------|---------------|-----------|------------------|--------------------|---------|
| drug | 39 ([drug #drug-model][drug-model] + 入れ子クラス合算で約 113) | 4 階層 (`dosage → renalAdjustment[] → range → severity`) | pharmacokinetics 500-1,250 字 + clinicalResults 300-2,000 字 | 5-10 KB | 120 件で 0.6-1.2 MB |
| disease | 25 ([disease #disease-model][disease-model] + 入れ子合算で約 66) | 3 階層 (`treatments → pharmacological[] → drugIds[]`) | etiology 150-500 字 + summary 150-300 字 | 3-5 KB | 80 件で 0.24-0.4 MB |

フル entity 200 件合計 0.84-1.6 MB。本プランは詳細エンティティをブックマーク DB に保存せず、リスト最小項目 (snapshot) のみ保存する。

### 検索軸フィールド列挙

検索画面の絞り込み軸で使う医薬品/疾患モデルのフィールド一覧。

**医薬品** ([drug][drug]):

| 軸種 | フィールド | 型 |
|------|-----------|----|
| キーワード | `genericName` / `brandName` / `brandNameKana` / `therapeuticCategoryName` / `composition.activeIngredient` / `manufacturer` | String |
| 絞り込み (共通基盤定義済) | `dosageForm` (13 値) / `routeOfAdministration` (8 値) / `regulatoryClass` (11 値) / `atcCode` 第 1 階層 (14 群) / `therapeuticCategory` (enum constant 名) | Enum / List<Enum> |
| 絞り込み (副作用・併用注意・特定背景) | `adverseReactions.serious[].name` / `frequency` (4 値) / `other.over5Percent[]` / `interactions.combinationProhibited[].displayName` / `precautionsForSpecificPopulations[].category` (8 値) | String / Enum |
| ソート | `revisedAt` / `brandNameKana` / `atcCode` / `therapeuticCategoryName` | String |

**疾患** ([disease][disease]):

| 軸種 | フィールド | 型 |
|------|-----------|----|
| キーワード | `name` / `nameKana` / `nameEnglish` / `synonyms[]` | String / List<String> |
| 絞り込み (共通基盤定義済) | `icd10Chapter` (22 章) / `medicalDepartment` (16 科) / `chronicity` (4 値) / `infectious` (bool) | Enum / List<Enum> / bool |
| 絞り込み (症状・発症・検査・治療有無) | `symptoms.mainSymptoms[]` / `onsetPattern` (5 値) / `requiredExams[].category` (5 値) / `severityGrading` null 判定 / `treatments.pharmacological[]` 非空判定 | String / Enum |
| ソート | `revisedAt` / `nameKana` / `icd10Chapter` | String / Enum |

`atcCode` は 5 階層コードで格納 (正規表現 `^[ABCDGHJLMNPRSV][0-9]{2}[A-Z][A-Z][0-9]{2}$`) なのでグループ絞り込みは前方一致 (`category_atc` クエリは先頭 1 文字)。`regulatoryClass` と `medicalDepartment` は `List<Enum>` 型のためフィールド側の包含判定で OR 結合。

**共通仕様**:

- 複数指定可のフィルタ (`regulatory_class`, `dosage_form`, `route`, `precaution_category`, `onset_pattern`, `exam_category`) は **OR 結合**。AND 結合が必要な場合は別途要件追加
- 単一指定のフィルタ (`icd10_chapter`, `department`, `chronicity`) は値一致のみ
- キーワード欄と絞り込みの組合せは AND (キーワード ∧ フィルタすべて ∧ ソート)
- **キーワード検索発火タイミング**: 画面内の検索ボタン押下、またはソフトキーボードの送信アクションキー押下 (Flutter `TextField.onSubmitted` / iOS `onSubmit`) のいずれか。debounce 入力中の Live 検索は対象外 (1 文字入力ごとに `search_history` が入力途中文字列で埋まる履歴汚染と、`total_count` バッジ・結果リストが入力 1 文字ごとに大幅に変動する UX 不安定性を避けるため)
- 絞り込み条件 (フィルタ / ソート / 一致モード / キーワード対象種別) を 1 つでも変更した時点で結果リストをリセットしページング初期化、即時再検索を発火
- 結果リストの pull-to-refresh は実装しない (pull-to-refresh は `/v1/categories` フィルタマスタ再取得のみ対象、運用・非機能要件参照)
- 検索履歴の件数上限・重複排除・タブ分離は [search-history][sh] の仕様に従う

## API エンドポイント仕様

参照型アプリのため HTTP 動詞は GET のみ。書き込み (ブックマーク / 履歴 / 計算結果) はローカル DB 完結で API 経由しない。実装側仕様は [base #api-spec][base-api] / [base #image-endpoints][base-img] に集約済。本セクションはアプリ側からの利用視点のサマリ。

### エンドポイント一覧

### 共通仕様

- **Base URL**: 別セッションで確定 (モックサーバー URL)。クライアント定数として `lib/config/api_config.dart` / `Sources/APIConfig.swift` 等に集約 (基本方針「SSOT 遵守」)
- **API バージョンプレフィックス**: 全データ系・画像系エンドポイントは `/v1/` 配下
- `Content-Type: application/json; charset=utf-8` (画像系は `image/png`)
- 認証: なし (ポートフォリオ公開モック)
- ページング envelope: `{ items, page, page_size, total_pages, total_count, disclaimer }` ([base #api-spec][base-api])
- 空値ポリシー: `List<T> = []` / `T? = null` 両方許容 ([base #type-conventions][base-types])
- エラーレスポンス: HTTP 4xx/5xx でも body は JSON `{ code, message, details, disclaimer }` ([base #api-spec][base-api])
- タイムアウト / リトライ: 接続 10 秒 / 読み込み 15 秒 / 5xx のみ 1 回リトライ (指数バックオフ 500ms)
- root `disclaimer` フィールド: `Drug` / `Disease` / `DrugListResponse` / `DiseaseListResponse` の root に必ず付与。`CategoriesResponse` には付かない ([base #disclaimer-field][base-disc])。クライアントは画面下部の小ラベルや About 画面で参照可能

## 運用・非機能要件

### 永続化設計

永続化対象 5 種類のスキーマ定義と使用箇所・ライフサイクルは、それぞれ別ファイルに委譲する。本セクションでは構成と横断的方針のみ扱う。

| 永続化対象 | 種別 | 親画面 | スキーマ定義 |
|----------|------|-------|--------------------|
| ブックマーク | RDB テーブル | ブックマーク一覧 / 詳細画面トグル | [bookmark][bm] |
| 閲覧履歴 | RDB テーブル | 閲覧履歴画面 / 詳細画面 (自動追加) | [browsing-history][bh] |
| 計算履歴 | RDB テーブル | 計算ツール画面 (折りたたみセクション) | [calculation-history][ch] |
| 検索履歴 | RDB テーブル | 検索画面 (キーワード欄フォーカス時ドロップダウン) | [search-history][sh] |
| テーマ設定 | key-value | UI 切替手段なし、システム追従のみ | [theme-settings][th] |

**横断方針**:

- 実装技術: Flutter は [drift][drift]、iOS は [SwiftData][swiftdata] (技術スタック節参照)
- マイグレーション戦略: 技術スタック対称表の「マイグレーション戦略」行参照
- バックアップ / エクスポート: 対象外 (ポートフォリオ目的、ローカル完結)
- 起動時の整合性チェック: ブックマーク・閲覧履歴の snapshot と API レスポンスは自動 sync (詳細画面遷移時に更新)

**ブックマークのネットワーク戦略** (スナップショット + オンラインフォールバックは全永続化対象の中でブックマークのみ):

1. 詳細画面遷移時、常に `/v1/drugs/{id}` / `/v1/diseases/{id}` を呼ぶ
2. 成功: 最新データで詳細表示、ブックマークされていれば `snapshot` を上書き更新 (自動 sync)
3. 失敗: ブックマーク済なら `snapshot` 最小項目のみ表示 + 警告バナー「オフライン表示中。最新情報は接続時に取得されます」+ 再試行ボタン。未ブックマークならエラー画面

### ページネーション

共通基盤の page-based envelope 仕様 ([base #api-spec][base-api]) + クライアント無限スクロール。

- クライアント変数 `pageSize = 20` / HTTP クエリ `page_size=20` (mock-server `SearchDefaults.DEFAULT_PAGE_SIZE` と一致)
- UI: スクロール末端検出で自動次ページ取得 (infinite scroll)。検索条件変更時はページリセット
- 最終ページ判定: `page >= envelope.total_pages` またはレスポンス `items.length < page_size` で打ち切り (envelope に `has_more` フラグなし、自前判定)
- 初期ロード時 skeleton UI ([skeletonizer][skeleton-flutter] / [redacted][redacted])
- 2 ページ目以降のロード時はリスト末尾に小サイズスピナー

### ローディング表示

UIState 4 状態 enum (`Loading | Success | Error | Empty`) + プラットフォーム標準 Skeleton UI を採用。

**共通の UI State enum** (Contract Pattern):

```
UIState<T>:
  | .loading            // 初回ロード (skeleton)
  | .success(T)          // 正常データ
  | .loadingMore(T)      // 次ページロード中 (既存データ表示維持 + スピナー)
  | .error(AppError)     // エラー (下記)
  | .empty               // 結果0件 (下記)
```

Flutter では sealed class、iOS では enum で定義。画面ロジックは全てこの enum を switch する。

**Flutter 側の実装方針**:

- 状態管理: Riverpod の `AsyncValue<T>` または Freezed sealed class で `Loading / Data / Error / Empty` を表現
- 初回ロード: [skeletonizer][skeleton-flutter] で行プレースホルダ表示
- 追加ロード (ページネーション): リスト末尾に `CircularProgressIndicator.adaptive()` の小サイズ
- 全画面待機 (短時間処理): 即時完了扱い、300ms 超で初めてオーバーレイ表示

**iOS 側の実装方針**:

- 状態管理: TCA Reducer の `state.loading: LoadingState` enum / `@Observable` の場合は `enum ViewState`
- 初回ロード: [.redacted(reason: .placeholder)][redacted] 標準 API で骨格表示
- 追加ロード: `ProgressView().controlSize(.small)` リスト末尾
- 全画面待機: `ProgressView("読み込み中")` オーバーレイ

### エラー・Empty 表示

**エラー種別と表示**:

| エラー種別 | 検出条件 | メッセージ例 | アクション |
|-----------|---------|-------------|----------|
| ネットワーク断 | HTTP Timeout / Socket Error | 「通信できません。ネットワーク接続を確認してください。」 | 再試行ボタン |
| サーバエラー (5xx) | HTTP 500-599 | 「一時的な問題が発生しました。しばらく経ってから再度お試しください。」 | 再試行ボタン |
| リクエストエラー (4xx) | HTTP 400-499 | API の `message` を表示 (例: 「該当するデータが見つかりません」) | 戻る / 条件変更 |
| パース失敗 | Codable/Freezed 失敗 | 「データを読み込めません。」 | 再試行 + 開発者通知 (`OSLog` / `developer.log`) |

API エラーレスポンス構造は [base #api-spec][base-api] に準拠。`code` 値ごとに上記表のエラー種別にマッピング。`code` の取り得る値: `BAD_REQUEST` / `NOT_FOUND` / `INVALID_THERAPEUTIC_CATEGORY` / `INVALID_PRECAUTION_CATEGORY` / `INVALID_ONSET_PATTERN` / `INVALID_EXAM_CATEGORY` / `INVALID_SORT_KEY` / `INTERNAL_ERROR`。

**タイムアウト・リトライ設定**:

- HTTP 接続タイムアウト: 10 秒
- HTTP 読み込みタイムアウト: 15 秒
- リトライ: サーバエラー (5xx) のみ 1 回自動リトライ、指数バックオフ (500ms)。4xx / パース失敗はリトライなし

**Empty 表示**:

| 画面 | Empty 条件 | 表示内容 | 次のアクション |
|------|----------|---------|---------------|
| 検索結果 | envelope `total_count == 0` | アイコン + 「該当するデータがありません」 | 「条件を変更する」ボタン (フィルタリセット) |
| 検索履歴ドロップダウン | 該当 `target` の履歴 0 件 | ドロップダウンを非表示 | (表示せず) |
| ブックマーク (タブ別) | 該当 `id` 接頭辞のブックマーク 0 件 | 「ブックマークがありません」+ 使い方ヒント | 「検索画面へ」ボタン |
| 閲覧履歴 (タブ別) | 該当 `id` 接頭辞の閲覧履歴 0 件 | 「閲覧履歴がありません」 | 「検索画面へ」ボタン |
| 計算履歴 (calcType 別) | 該当 `calcType` の履歴 0 件 | 「計算履歴がありません」 | (折りたたみを展開せずに閉じたまま) |

エラー・Empty は上記 UIState enum の一部として実装され、画面コンポーネントは enum を switch することで統一的に扱う。
