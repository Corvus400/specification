# CRUD WebAPI 要件定義

本書は、架空医薬品・疾患リファレンス backend に追加する **医薬品 (drug) / 疾患 (disease) の CRUD WebAPI の入出力契約と振る舞い要件 (WHAT)** を定義する。実現手段 (HOW) は本書では規定せず、実装フェーズで別途設計する。

関連文書:

- アクセス制御 (到達経路の分離・外部秘匿・認証/認可・CORS・OpenAPI 除外) は `13_秘匿アクセス制御要件定義.md` で定義する。本書は **同書が秘匿対象とする書き込み系 API の中身**を規定し、path・メソッド割付・認可は 13_ を前提とする。
- フィールドの完全な意味論・値域は既存のモデル詳細 `03_医薬品モデル詳細.md` / `04_疾患モデル詳細.md` に従う。本書はそれらを再定義せず、CRUD の観点 (入力可否・検証・サーバ管理) からの差分のみを規定する。
- CMS (クライアント側) の機能・UI 要件は `15_CMS要件定義.md` で定義する。

前提:

- 既存の読み取り専用 API (`GET /v1/drugs`・`GET /v1/drugs/{id}`・`GET /v1/diseases`・`GET /v1/diseases/{id}`) は公開面に維持され、その応答契約を変更しない (回帰禁止)。本書が新規に定義するのは Create / Update / Delete と画像アップロードである。
- データは実 DB に永続化され、drug / disease エンティティ全体が 1 レコードに直列化格納される。外部公開 ID とは別に DB 内部の主キーが存在する (本書 §3)。

---

## 1. 目的

- drug / disease に対する Create / Read / Update / Delete を、既存の読み取り契約と整合する形で提供する。
- クライアント入力が許される範囲とサーバが管理する範囲を明確に分離し、不正な値・不整合な参照・編集の競合を検出して拒否する。
- drug 画像のアップロード・保存・配信を、既存の画像配信契約を壊さずに追加する。
- 編集結果の永続性と、初期投入データ (seed) との非対称性を運用要件として明示する。

---

## 2. 対象リソースと操作

| リソース | 外部公開 ID 形式 | 対象操作 |
|---|---|---|
| 医薬品 (drug) | `drug_NNNN` | Create / Read (一覧・詳細) / Update / Delete / 画像アップロード |
| 疾患 (disease) | `disease_NNNN` | Create / Read (一覧・詳細) / Update / Delete |

- path・HTTP メソッド・到達経路・認証/認可は `13_秘匿アクセス制御要件定義.md` §2・§4 が規定する (書き込み系は管理面に分離し、外部からは存在秘匿)。本書では操作の入出力契約のみを定義する。
- Read (一覧・詳細) は既存の公開 API と同一契約とし、本書では回帰禁止の対象として参照するに留める (§4.2)。
- 疾患には画像アップロードを設けない (画像を持つのは drug のみ)。

---

## 3. サーバ管理フィールド (クライアント入力不可)

以下はサーバが値を決定・維持するフィールドであり、Create / Update のリクエストボディで**クライアントが指定できない**。クライアントが送出した場合は無視するか検証エラーとする (いずれを採るかは実装で選ぶが、「クライアント値が永続化されない」ことを要件とする)。

| フィールド | 対象 | 振る舞い |
|---|---|---|
| 外部公開 ID (`id`) | drug / disease | Create 時にサーバが採番する (`drug_NNNN` / `disease_NNNN`)。採番後は immutable で、Update で変更できない (path で対象を識別する不変キー)。 |
| `revisedAt` | drug / disease | サーバが管理する改訂日 (`YYYY-MM-DD`)。Create 時にサーバが設定し、Update 時に当日へ自動更新する (§7)。 |
| `imageUrl` | drug | `id` と `dosageForm` から派生する導出値。クライアントは設定できない。drug 画像アップロードの有無と現在の `dosageForm` を反映する (§6)。 |
| `disclaimer` | drug / disease | 定数の注意書き。クライアントは設定できない。 |
| 版管理タイムスタンプ (`created_at` / `updated_at` 相当) | drug / disease | DB が管理する作成/更新時刻。クライアントは設定できない。`updated_at` 相当は競合検出の版識別子として用いる (§7)。 |

補足 (ID の二重構造):

- クライアントが目にする ID は外部公開 ID (`drug_NNNN` / `disease_NNNN`) のみである。DB には別途、外部に露出しない内部主キーが存在する。両者を混同しない。
- 外部公開 ID はサーバ採番・以後不変であり、クライアントが Create 時に任意指定することはできない。

## 4. 入出力契約

### 4.1 共通

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CRUD-1 | コンテンツフィールドの範囲 | Create / Update のリクエストボディで指定できるのは、対象モデル (`03_` / `04_`) のフィールドのうち §3 のサーバ管理フィールドを**除いた**全フィールド (本書では「コンテンツフィールド」と呼ぶ)。 |
| CRUD-2 | レスポンス形状の一致 | Create / Update の成功応答は、Read 詳細 (既存 `GET /v1/{resource}/{id}`) と**同一形状の完全なエンティティ**を返す。クライアントが Read で受け取る形と書き込み後に受け取る形を一致させる。 |
| CRUD-3 | コンテンツ型 | リクエスト/レスポンスの本文は JSON とする。フィールドの直列化名・enum 値・nested 構造は既存モデル契約 (`03_` / `04_`、既存 `GET` 応答) と一致させる。 |

### 4.2 Read (回帰禁止)

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CRUD-4 | 一覧契約の不変 | `GET /v1/drugs`・`GET /v1/diseases` の応答 (サマリ配列 `items` とページング情報 `page` / `pageSize` / `totalPages` / `totalCount` / `disclaimer`、および各種フィルタ・ソート・キーワード) を変更しない。CRUD で編集されたレコードも同一の一覧契約で返す。 |
| CRUD-5 | 詳細契約の不変 | `GET /v1/drugs/{id}`・`GET /v1/diseases/{id}` の応答フィールド集合・存在しない ID に対する 404 挙動を変更しない。 |

### 4.3 Create

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CRUD-6 | 入力 | コンテンツフィールド (§4.1 CRUD-1) を必須/任意の別 (§5) に従って受け取る。サーバ管理フィールド (§3) は受け付けない。 |
| CRUD-7 | ID 採番 | サーバが新規の外部公開 ID (`drug_NNNN` / `disease_NNNN`) を採番する。既存 ID と衝突しない一意な値であること。 |
| CRUD-8 | 出力 | 採番済み ID・サーバ設定済み `revisedAt`・導出済み `imageUrl` (drug) を含む完全なエンティティを返す (CRUD-2)。 |
| CRUD-9 | 検証失敗 | §5 の検証に違反する入力は永続化せず、検証エラー (§9) を返す。 |

### 4.4 Update

drug / disease の更新は **全置換** と **部分更新** の双方を提供する (13_ §2 が管理面に `PUT`・`PATCH` を割り当てていることと整合)。いずれの方式でも外部公開 ID は変更できず (§3)、`revisedAt` / `updated_at` 相当はサーバが自動更新し (§7)、競合検出 (§7) の対象とする。

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CRUD-10 | 全置換 | 全置換は、対象リソースのコンテンツフィールド一式を受け取り、欠落した任意フィールドはその既定 (空・未設定) に戻す。必須フィールド (§5) の欠落は検証エラー。 |
| CRUD-11 | 部分更新 | 部分更新は、提示されたフィールドのみを更新し、提示されなかったフィールドは現在値を維持する。提示された各フィールドは §5 の検証を受ける。 |
| CRUD-12 | ID 不変 | リクエストボディに外部公開 ID を含めても対象の ID は変更されない。対象は path の ID で識別する。 |
| CRUD-13 | 対象不在 | 存在しない ID への更新は 404 (§9)。 |
| CRUD-14 | 出力 | 更新後の完全なエンティティ (自動更新後の `revisedAt` を含む) を返す (CRUD-2)。 |

### 4.5 Delete

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CRUD-15 | 削除 | 指定 ID のリソースを削除する。成功応答は本文なし、または削除済みを示す応答とする。 |
| CRUD-16 | 冪等性 | 同一 ID への複数回の削除で副作用を生じない。2 回目以降は新たな変更を起こさず、削除済み (または不在) として一貫した結果を返す。 |
| CRUD-17 | 被参照リソースの削除 | 削除対象が他リソースから参照されている場合 (例: ある disease の関連薬として参照される drug) は、削除を**拒否**し競合 (§9、409) を返す。これは「ダングリング参照を生まない」不変条件 (§5 CRUD-21) を削除操作でも維持するためである。参照を外してから削除する運用とする。 |
| CRUD-18 | 画像の連動 | drug 削除時、その drug に紐づくアップロード済み画像 (§6) も到達不能とする (孤立した画像を残さない)。既存の剤形画像・既定画像 (override 対象を除く) は drug に固有でないため影響を受けない。 |

## 5. 検証と参照整合性

エラーは RFC 9457 problem+json (§9) で統一して返す。フィールド単位の違反は `errors[]` (フィールド名・理由) に列挙する。

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CRUD-19 | 必須フィールド | モデル契約 (`03_` / `04_`) で必須 (非 null・既定なし) とされるフィールドの欠落は検証エラー。任意フィールドの省略は許容する。 |
| CRUD-20 | enum 値域 | enum 型フィールド (例: 剤形・規制区分・投与経路・ICD-10 章・慢性度・診療科 等) は、既存モデルが定義する許容値 (直列化名) のみを受け付ける。未知値は検証エラー。 |
| CRUD-21 | 参照整合性 (ダングリング禁止) | `relatedDrugIds` / `relatedDiseaseIds` に列挙する ID は、すべて実在するリソースを指すこと。存在しない ID (ダングリング参照) を含む入力は検証エラーとして拒否する。 |
| CRUD-22 | 自己参照禁止 | リソースが自身の ID を自身の関連 ID リストに含めることを禁止する (検証エラー)。 |
| CRUD-23 | 形式検証 | ID 形式 (`drug_NNNN` / `disease_NNNN`)、日付形式 (`YYYY-MM-DD`)、その他モデルが定める形式制約に違反する値は検証エラー。ただしサーバ管理フィールド (§3) はクライアント検証の対象外 (入力を受け付けないため)。 |

## 6. drug 画像アップロード

既存の drug 画像配信は、アプリに同梱された読み取り専用リソースからの配信であり、ごく一部の drug についてのみ固有画像が存在する。それ以外の drug は剤形に対応する既定画像へフォールバックする。本要件は、任意の drug に固有画像をアップロードして差し替えられるようにする。

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CRUD-24 | アップロード | 指定 drug にバイナリ画像をアップロードできる。対象 drug が存在しない場合は 404。 |
| CRUD-25 | 書き込み可能な保存先 | アップロード画像は、アプリ同梱の読み取り専用リソースとは別の、**実行時に書き込み可能な永続保存先**に保存する (同梱リソース領域は実行時書き込み不可のため)。 |
| CRUD-26 | 配信の解決 | drug 画像配信は、アップロード済み画像が存在する drug についてはそれを返す。存在しない drug は従来どおり剤形の既定画像にフォールバックする。**既存の固有画像を持つ drug の配信応答は変更しない (回帰禁止)**。 |
| CRUD-27 | `imageUrl` の整合 | drug の `imageUrl` (§3、導出値) は、固有画像 (同梱 or アップロード) を持つ drug についてはその drug 固有の画像配信先を指し、持たない drug は剤形既定画像を指す。固有画像の有無は**静的な固定集合ではなく実データに基づいて**判定すること (アップロードにより固有画像を持つようになった drug は、その固有配信先を指すこと)。 |
| CRUD-28 | 形式・サイズ制約 | 受け付ける画像形式は PNG とする。1 ファイルあたりのサイズ上限を設ける (具体値は実装で定める)。上限超過・非対応形式は検証エラー (§9)。 |
| CRUD-29 | パストラバーサル防止 | 保存先パスは drug の外部公開 ID (`drug_NNNN` 形式検証済み) からサーバが構築し、クライアント入力をパスに連結しない。`..` 等によるディレクトリ脱出を不可能にする。 |
| CRUD-30 | リサイズ整合 | アップロード画像も既存配信と同じサイズ指定 (`S` / `M` / `Original`) で取得できること。 |

## 7. 改訂日・タイムスタンプの自動更新と競合検出

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CRUD-31 | `revisedAt` 自動 bump | Update (全置換・部分更新・画像アップロードを含む内容変更) の成功時、サーバが `revisedAt` を当日 (`YYYY-MM-DD`) に自動更新する。クライアントは `revisedAt` を指定できない (§3)。 |
| CRUD-32 | タイムスタンプ自動更新 | DB の更新時刻 (`updated_at` 相当) はサーバが各更新で自動更新する。 |
| CRUD-33 | 楽観的同時実行制御 | Update / Delete は楽観ロックを要件とする。クライアントが既知の版 (`updated_at` 相当または ETag、いずれを採るかは実装で選ぶ) を提示し、サーバ側の現在版と一致する場合のみ適用する。 |
| CRUD-34 | 競合時の応答 | 提示された版が現在版と一致しない場合は適用せず、競合として拒否する。前提条件 (If-Match 等) の不一致は 412、版フィールド不一致は 409 を返す (いずれも RFC 9457 problem+json)。これにより複数編集の取り違え・上書き消失を防ぐ。 |

## 8. レート制限・OpenAPI 除外 (13_ への参照)

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CRUD-35 | OpenAPI 除外 | 本書が定義する書き込み系 API・画像アップロードの path とスキーマは、公開 OpenAPI ドキュメントに含めない (`13_秘匿アクセス制御要件定義.md` SEC-9)。 |
| CRUD-36 | レート制限の一貫性 | 管理面に置かれる本 API のレート制限方針は、既存の管理エンドポイント (`/v1/admin/*`) の方針と一貫させる。公開面の既存読み取り API のレート制限挙動は変更しない。 |

## 9. エラー契約

すべての失敗応答は RFC 9457 problem+json で返す (既存 `ProblemDetails`・`FieldViolation`・`DomainError` の契約に従う。詳細は既存のエラー契約仕様を参照)。

| 失敗 | ドメインエラー区分 | HTTP ステータス |
|---|---|---|
| 対象リソース不在 (Update / Delete / 画像対象 drug) | NotFound | 404 |
| 入力検証違反 (必須・enum 値域・形式・自己参照、参照整合性違反 §5) | Validation | 422 (`errors[]` に違反フィールドを列挙) |
| 編集競合 (版不一致)・被参照リソースの削除 (§4.5 CRUD-17) | Conflict | 409 |
| 前提条件不一致 (If-Match 等、§7 CRUD-34) | — | 412 |
| 未認証 / 権限不足 | Unauthorized / Forbidden | 401 / 403 (`13_` §4.2 に従う。外部経路での秘匿は `13_` SEC-2 が規定し、本書の認証/認可セマンティクスはローカル到達後に適用される) |
| 想定外エラー | Unexpected | 500 |

- 検証エラーの `errors[]` には、違反したフィールド名と理由を機械可読に列挙する。
- 認証・認可に関する応答 (401/403) と外部からの存在秘匿 (404) の切り分けは `13_秘匿アクセス制御要件定義.md` の規定に従い、本書では重複定義しない。

## 10. 運用要件 (初期投入データとの非対称性)

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| CRUD-37 | 編集の永続性 | CRUD による編集結果は DB に永続化され、以後の Read に反映される。 |
| CRUD-38 | seed との非対称性の明示 | 初期投入データ (DB マイグレーションの seed) は**新規 DB に対して一度だけ**適用される。CRUD による編集は seed 定義 (マイグレーション SQL) へは**反映されない (round-trip しない)**。したがって DB を破棄して再作成すると CRUD 編集は失われ、seed 時点の内容に戻る。この非対称性を運用上の既知の制約として明記し、編集内容の保全はバックアップ等の運用で担保する (具体手段は実装/運用フェーズ)。 |

## 11. 受け入れ確認の観点

本書は要件定義のため実行コードを伴わない。以下のランタイム検証は実装フェーズで行う (各 acceptance criteria に対応)。

| 観点 | 期待 | 対応要件 |
|---|---|---|
| Create → Read | 採番された ID で詳細取得でき、書き込み時と同形状のエンティティが返る | CRUD-2, CRUD-7, CRUD-8 |
| サーバ管理フィールドの入力 | クライアント指定値が永続化されない | CRUD-1, §3 |
| 全置換 / 部分更新 | 全置換は欠落任意フィールドを既定化、部分更新は未提示フィールドを維持 | CRUD-10, CRUD-11 |
| ダングリング参照・自己参照を含む入力 | 422 検証エラー (`errors[]` に該当フィールド) | CRUD-21, CRUD-22 |
| 被参照リソースの削除 | 409 競合で拒否 | CRUD-17 |
| 削除の冪等性 | 複数回 DELETE で副作用なし | CRUD-16 |
| 画像アップロード後の配信・`imageUrl` | 当該 drug の固有画像が返り、`imageUrl` が固有配信先を指す。既存固有画像 drug の応答は不変 | CRUD-26, CRUD-27 |
| 非対応形式・上限超過の画像 | 422 検証エラー | CRUD-28 |
| 版不一致の Update | 412 または 409 で拒否、上書き消失なし | CRUD-33, CRUD-34 |
| Update 後の `revisedAt` | 当日へ自動 bump | CRUD-31 |
| 既存 `GET /v1/*` の挙動 | 変更前と不変 (回帰確認) | CRUD-4, CRUD-5 |
| 公開 OpenAPI | 書き込み系 path・スキーマを含まない | CRUD-35 |
